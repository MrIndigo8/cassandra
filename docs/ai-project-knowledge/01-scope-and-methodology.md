# Область проверки и методология

## Запрос «проверить каждую строчку во всех файлах»

В репозитории **сотни исходных файлов** (TypeScript/TSX, SQL, JSON и др.). Полный построчный аудит всего кода человеком или одним сеансом ИИ **невыполним** как гарантия отсутствия дефектов: это требует статического анализа, покрытия тестами, ревью и эксплуатационных проверок.

### Что сделано в рамках этого набора документов

1. **Инвентаризация**: все маршруты API в `src/app/api/**/route.ts`, страницы App Router в `src/app/[locale]`, миграции в `supabase/migrations/`.
2. **Трассировка связей**: цепочка «запись → анализ → deep_analysis → fingerprint / верификация / снимки / уведомления» по чтению ключевых модулей и оркестратора cron.
3. **Автоматические проверки**: выполнены `npm run test` (Vitest) и `npm run build` (Next.js 14) — результат зафиксирован в [07-qa-build-health-risks.md](./07-qa-build-health-risks.md).
4. **Известные риски из правил проекта** (см. `.cursor/rules`): учтены в [05-database-schema-notes.md](./05-database-schema-notes.md) и [07-qa-build-health-risks.md](./07-qa-build-health-risks.md).

### Исключения из «полного» обзора

| Область | Причина исключения |
|---------|-------------------|
| `.claude/worktrees/**` | Дубликаты/черновики; **источник истины — корень проекта** без этого пути |
| `node_modules/` | Зависимости; не исходный код продукта |
| Секреты и значения `.env*` | Не документируются; только **имена** переменных окружения |

### Как углубить QA

- Запуск `npm run lint`, E2E (при появлении), нагрузочное тестирование API с `CRON_SECRET`.
- Проверка RLS в Supabase для всех таблиц, с которыми работает клиент.
- Ручной проход критических пользовательских сценариев по [06-ui-ux-inventory.md](./06-ui-ux-inventory.md).
